# CLI command catalog

This catalog separates bootstrap commands implemented in the current shell from roadmap-only commands.

The executable catalog is structural. Every command publishes `flags[]` and `positionals[]` metadata used by help, CLI parsing, and connected lifecycle validation. Unknown flags and duplicate non-repeatable flags fail before command-handler dispatch; transport payloads cannot widen the catalog.

Top-level help groups implemented commands by operator surface: guided shortcuts,
core lifecycle, run control, review/QA, delivery, release, and operations. This
keeps the command list scannable while preserving every existing grouped command
contract.

`aor project verify` writes one `step-result` per bounded command. Each command step result includes the command owner/source/kind, command-group id/role/phase/enforcement/timeout class, enforcement result, timeout status, start/finish timestamps, duration, exit code, signal/error code, bounded stdout/stderr excerpts, and a transcript evidence ref. Verify summaries aggregate command groups so required failures fail the run, warning groups produce warning evidence, and observe groups remain non-blocking evidence.
`aor project verify --plan` writes `verification-plan.json` or a label-specific `verification-plan-<label>.json` sidecar with generated command groups, discovery candidates, confidence, source refs, and the latest per-group verify status when available. The plan surface does not run target commands and is exposed through CLI JSON plus the API/web project-state read model.
Exit-0 commands that emit high-signal warning output on stderr are marked failed with bounded `output_quality_findings[]`. When `--output-quality-baseline` references prior verify summaries with matching warning classes, current findings stay visible as `baseline_status=pre_existing` evidence but do not fail the current command. The same flag also accepts an explicitly matching failed baseline command as `baseline_failure_status=pre_existing` evidence, so post-change verification can distinguish a new regression from an already broken target baseline.

## Guided first-run shortcuts for W21-S02/W21-S04/W31-S01
- `aor doctor`, `aor onboard`, `aor app`, and `aor next` are additive installed-user shortcuts. They do not remove or rename grouped command contracts.
- Guided shortcuts default to human-readable output. Pass `--json` to preserve the full machine-readable schema for scripts, tests, and control-plane wrappers. Pass `--json compact` for a populated-fields-only operator summary when the full compatibility schema is too noisy.
- `aor onboard <repo>` delegates to the existing `aor project init` bootstrap path. Clean repositories default to bundled asset mode, emit an onboarding report, and resolve project-profile registry roots without copying example registries into the target repo. Materialized asset mode remains explicit.
- `aor app` launches the optional packaged local SPA in a foreground loopback server. `--smoke --open false --json` starts the server, checks `/`, `/app-config.json`, `GET /api/projects`, `GET /api/projects/:projectId/state`, first-run wizard, project switcher, flow selector, and `New Flow` bundle markers, prints JSON, and exits for CI/release smoke.
- `aor mission create` is an additive guided wrapper over `aor intake create`; it preserves product goals, constraints, KPIs, Definition of Done, source refs, allowed paths, and delivery mode in the existing intake-request packet/body contracts.
- `aor next` writes a deterministic `next-action-report` with exactly one primary action, explicit blockers, evidence refs, write-back policy, and final-stage closure state before any delivery-capable recommendation.

## Operator semantics for W5-S03
- `aor run status`, `aor packet show`, and `aor evidence show` are read-only operator commands.
